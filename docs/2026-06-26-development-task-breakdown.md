# 求职工作台开发任务拆解清单

## 目标

把当前“可演示 MVP”推进到“可持续开发的产品骨架”。

推进顺序：

1. 先补核心闭环稳定性
2. 再补真实 AI 与数据持久化
3. 然后做产品可用性升级
4. 最后做上线与商业化准备

---

## 阶段一：核心闭环加固

### 1.1 文件解析链路

- [ ] 校验 `.pdf/.doc/.docx` 上传类型与大小限制
- [ ] 统一上传失败提示文案
- [ ] `.doc` 解析失败时返回更友好的 fallback 提示
- [ ] 扫描版 PDF 明确提示“当前不支持 OCR”
- [ ] 上传成功后统一返回文件状态、预览状态、解析状态

### 1.2 单 JD 流程

- [ ] 校验“上传简历 -> 填写/抓取 JD -> 分析”全链路
- [ ] 统一 loading / success / fail 状态展示
- [ ] 补空状态、超时状态、接口失败状态
- [ ] JD 链接抓取失败时明确引导用户手动粘贴
- [ ] 报告区异常时保留已输入内容，不清空页面

### 1.3 海投流程

- [ ] 校验批量 JD 文本导入
- [ ] 校验批量 JD 链接导入
- [ ] 校验失败链接不阻断整批导入
- [ ] 校验聚类结果、版本草稿、导出占位链路
- [ ] 给失败项增加可读状态展示

### 1.4 前端状态整理

- [ ] 收拢 `public/app.js` 中单 JD、海投、报告派生逻辑
- [ ] 拆分“请求逻辑 / 状态派生 / DOM 渲染”三个层次
- [ ] 减少重复 DOM 更新逻辑
- [ ] 让空状态和结果状态共用统一渲染入口

### 阶段一验收

- [ ] 单 JD 和海投两条主流程都能稳定跑通
- [ ] 所有异常路径都有明确页面提示
- [ ] 没有“接口失败但页面无反馈”的情况

---

## 阶段二：真实 AI 与数据持久化

### 2.1 DeepSeek 正式接入

- [ ] 明确哪些能力走 DeepSeek：
  - [ ] 简历结构化总结
  - [ ] JD 结构化总结
  - [ ] 关键词提取
  - [ ] 单 JD 解释性评分
- [ ] API 不可用时自动 fallback 到本地规则
- [ ] 返回结果补 `provider`
- [ ] 返回结果补 `fallbackReason`
- [ ] 前端显示当前分析结果来源

### 2.2 数据库接入

- [ ] 引入 PostgreSQL
- [ ] 建立数据库连接层与环境变量配置
- [ ] 建立最小 migration 方案
- [ ] 第一版先不接 pgvector

### 2.3 最小数据模型

- [ ] `resume_records`
- [ ] `jd_records`
- [ ] `match_reports`
- [ ] `batch_runs`
- [ ] `jd_clusters`
- [ ] `resume_variants`

每张表至少包含：

- [ ] `id`
- [ ] `createdAt`
- [ ] `updatedAt`
- [ ] `status`
- [ ] 业务主字段

### 2.4 接口增强

- [ ] 保持现有 POST 接口不变
- [ ] 新增 `GET /api/resumes`
- [ ] 新增 `GET /api/jds`
- [ ] 新增 `GET /api/reports/:id`
- [ ] 新增 `GET /api/batch-runs/:id`
- [ ] 新增 `GET /api/resume-variants/:id`

### 阶段二验收

- [ ] 分析结果能保存
- [ ] 刷新页面后能回看
- [ ] 能区分 DeepSeek 结果和 fallback 结果

## 阶段三：产品可用性升级

### 3.1 历史记录

- [ ] 最近单 JD 分析列表

- [ ] 第一版先输出结构化 Word

### 3.5 JD 抓取体验

- [ ] 失败项列表化展示
- [ ] 支持逐条补文本
- [ ] 支持重试抓取单条失败 JD

### 阶段三验收
